一、起源与背景

"Harness" 一词来自马具——缰绳、马鞍、嚼子——是一套引导强大但不可预测的动物的完整装备。驾驭工程不是去削弱 AI 的能力，而是为它打造一套精密的控制系统，让它跑得又快又稳。

2026 年 2 月 5 日
HashiCorp 联合创始人 Mitchell Hashimoto 在博客中首次提出 "Harness Engineering" 术语，基于其在 Ghostty 终端模拟器项目中的 AI 辅助开发经验。
2026 年 2 月 11 日
OpenAI 发布《Harness Engineering: Leveraging Codex in an Agent-First World》——披露 3 人团队用 5 个月、零手写代码生成百万行代码的实验报告。
2026 年 2 月 17 日
Martin Fowler（经 Thoughtworks 工程师 Birgitta Böckeler）发表深度分析，将 Harness 归纳为三大分类框架。
2026 年 2 月 22 日
Charlie Guo（Artificial Ignorance）综合多团队实践发布 Playbook 级综述文章。
2026 年 3 月
Harness Engineering 成为全球开发者社区高频词，LangChain、Stripe、Anthropic 等团队纷纷分享实践经验。
核心理念

"Harness engineering is the idea that anytime you find an agent makes a mistake, you take the time to engineer a solution such that the agent will not make that mistake again in the future."

— Mitchell Hashimoto

二、核心定义

Harness Engineering（驾驭工程）是围绕 AI Agent 设计和构建约束机制、反馈回路、工作流控制和持续改进循环的系统工程实践。它不优化模型本身，而是优化模型运行的"环境"。

核心哲学

人类掌舵，智能体执行（Humans Steer, Agents Execute）

Martin Fowler 将其描述为"围绕 AI Agent 构建的工具与实践体系，用以确保其可控"——但好的 Harness 不仅让 Agent 更可控，也让 Agent 更有能力。

五、三大支柱详解

支柱一：上下文工程（Context Engineering）

核心命题

从 Agent 的视角，任何它在运行时无法访问的内容，等同于不存在。Google Docs、Slack 消息、工程师脑中的隐性知识——对 Agent 来说全是黑洞。

OpenAI 团队的关键经验是：巨型 AGENTS.md 文件是反模式。上下文是稀缺资源，过多的指导会挤占任务和代码的空间，变成"陈旧规则的坟场"。正确做法是将 AGENTS.md 作为"目录索引"（约 100 行），指向结构化的 docs/ 深层知识库。

知识库布局

repo/
├── AGENTS.md               # 根入口（~100行），全局规则
├── docs/
│   ├── architecture.md      # 架构地图：域与包分层
│   ├── design/              # 设计文档（含验证状态）
│   ├── plans/
│   │   ├── active/          # 进行中的执行计划
│   │   └── completed/       # 已完成计划（保留决策日志）
│   ├── quality.md           # 每个域/层的质量评分
│   └── tech-debt.md         # 已知技术债追踪
├── src/
│   └── feature-x/
│       └── AGENTS.md        # 子目录级别的局部规则
└── ...                      # OpenAI 仓库含 88 个 AGENTS.md
实践要点

渐进式披露：Agent 从小而稳定的入口出发，根据当前任务按需检索更深层上下文。Hashimoto 的 Ghostty 项目中，AGENTS.md 里每一行都对应一个历史 Agent 失败案例——文档是活的反馈循环，不是静态制品。

支柱二：架构约束（Architectural Constraints）

仅靠文档记录规则是不够的——如果不能机械化地强制执行，Agent 就会偏离。Agent 极其擅长模式复制，它会忠实地复制仓库中已有的任何模式——包括坏模式。

约束机制的关键设计

自定义 Linter 错误信息是 OpenAI 报告中被广泛称赞的巧妙设计：当 Agent 违反架构约束时，错误消息不仅标记违规，还告诉 Agent 如何修复。工具在 Agent 工作的同时"教会"了它。

# 示例：自定义 Linter 输出
ERROR: 违反层级依赖规则
  文件: src/settings/ui/SettingsPanel.tsx
  问题: UI 层直接导入了 Repo 层的 UserRepository
  修复: UI 层只能通过 Service 层访问数据。
        请改用 SettingsService.getUser() 方法。
        参考: docs/architecture.md#dependency-rules
关键悖论

约束解放创造力。限制 Agent 的解决方案空间，减少了在死胡同上浪费的 token，反而提升了整体产出质量和速度。就像高速公路上的护栏——正是因为有护栏，你才敢踩到 120 码。

支柱三：熵管理 / 垃圾回收（Garbage Collection）

Agent 快速生成大量代码，但文档、注释和架构图往往无法同步更新，代码库随时间"腐烂"。OpenAI 将技术债视为"高息贷款"——不能等问题严重时才集中处理，而是持续小额偿还。

具体做法包括：运行专门的"文档园丁 Agent"（Doc-gardening Agent）在后台扫描文档与代码的不一致，发现过时内容就自动提交 PR 修复。定期运行后台 Codex 任务扫描架构偏差、更新质量评分、发起针对性重构 PR。清理吞吐量与代码生成吞吐量成正比扩展。

传统开发时间

2025 年 8 月底，第一行代码提交到空仓库。初始脚手架（仓库结构、CI 配置、格式规则、包管理、应用框架）由 Codex CLI + GPT-5 基于模板生成。5 个月后，仓库包含约 100 万行代码，覆盖应用逻辑、基础设施、工具、文档和内部开发工具。全程零人工手写代码。

关键发现

布鲁克斯法则被打破：团队从 3 人扩展到 7 人后，吞吐量不但没有因为沟通成本增加而下降，反而继续增长。每位工程师日均处理 3.5 个 PR。这说明 Harness 模式具备与传统团队完全不同的扩展性。

规划是新的编码：工程师的工作不再是写代码，而是把模糊的大目标拆解成 Agent 能执行的小任务序列。当 Agent 卡住时，正确的问题不是"怎么让它更努力"，而是"缺少什么能力，以及如何让这个能力对 Agent 清晰可读"。

环境设计决定成败：团队最困难的挑战集中在设计环境、反馈回路和控制系统方面。代码本身变得廉价且可快速生成，真正的稀缺资源变成了设计可委托系统的能力。

注意事项

OpenAI 团队坦承，该实验是在全新代码库（greenfield）上进行的。将这套方法应用到有 10 年历史、缺少架构约束和文档的棕地项目（brownfield），是一个完全不同且尚未解决的挑战——如同在从未使用过静态分析工具的代码库上首次运行，会被告警淹没。

八、关键实证案例

案例一：Can Boluk — Hashline 格式实验

安全研究员 Can Boluk 仅改变了 Agent 的代码编辑格式（从传统 patch 改为 Hashline 格式），就在 16 个模型上显著提升了编码基准分数。最显著的 Grok Code Fast 1 从 6.7% 跃升至 68.3%——没有修改任何模型权重。

启示

一个格式的改变，等于十个模型升级。决定结果好坏的最大变量，往往不是模型有多聪明，而是模型被放在了一个什么样的环境里。

案例二：LangChain — Terminal Bench 2.0

LangChain 的编码 Agent 通过仅优化 Harness（文档结构、Build-Verify 循环、追踪系统），在 Terminal Bench 2.0 上得分从 52.8% 飙升至 66.5%，排名从全球第 30 位跃升至第 5 位。底层模型一个参数都没改。

案例三：Nicholas Carlini — C 编译器项目

Anthropic 工程师 Carlini 使用 16 个并行 Claude 实例，跨 2000+ 个会话构建了一个 10 万行 Rust C 编译器。关键实践包括：专门化 Agent（编译器工作、去重、性能、文档）、最小化控制台输出以减少上下文污染、确定性测试子采样（每个 Agent 随机运行 1-10% 的测试，集体覆盖完整测试套件）。

案例四：Stripe — Minions 系统

Stripe 内部部署了无人值守 Agent 系统 Minions，实现了每周千级 PR 的生产吞吐，独立验证了 Harness Engineering 在大规模生产环境中的可行性。

九、更多实践来源

Anthropic — 长时运行 Agent 的有效驾驭

Anthropic 工程团队发布了关于跨上下文窗口连续性的系统性研究，提出了初始化 Agent + 编码 Agent 双阶段方案。核心发现：即使是 Opus 级别的前沿模型，如果只给一个高层 prompt（如"构建一个 claude.ai 克隆"），跨多个上下文窗口也无法构建出生产级应用。

解决方案是将 Agent 分为两个角色：初始化 Agent 负责建立环境、写出完整的功能需求文件（expanding on the user's initial prompt），生成 claude-progress.txt 进度文件；后续的编码 Agent 每次启动时读取 git 历史和进度文件，快速理解工作状态，从上次中断处继续。这一模式让 Agent 在全新上下文窗口中也能有效恢复工作。

Anthropic 关键洞察

模型无法可靠评估自身工作。生产级 Harness 需要分离 Generator（生成者）和 Evaluator（评估者），类似 GAN 的对抗结构——一个 Agent 写代码，另一个 Agent（甚至同一模型的不同实例）来审查。

HumanLayer — 偏向发布，按需驾驭

HumanLayer 团队分享了一种务实的 Harness 策略：不要过度预优化，而是 "bias towards shipping"——只在 Agent 实际失败时才工程化修复。他们的 CLAUDE.md 控制在 60 行以内。

关键实践包括：验证机制必须是上下文高效的。早期他们让 Agent 每次修改后跑完整测试套件，4000 行通过的测试输出会淹没上下文窗口，Agent 因此丢失任务焦点开始产生幻觉。后来改为"成功静默，只输出错误"——大幅提升了可靠性。他们还发现过度细分 sub-agent 的工具权限反而导致 tool thrash，效果更差。

Trail of Bits — 安全审计领域的 Harness

知名安全公司 Trail of Bits 开源了他们的 Claude Code 配置体系，展示了 Harness 在安全审计领域的应用。核心做法包括：通过 Plugin Marketplace 分发专门的安全审计 Skills（漏洞分析、fuzz harness 生成、智能合约审计等），使用 Hooks 在 Agent 执行前注入安全检查，以及每周运行 /insights 分析会话模式来持续优化 Harness。

他们特别强调"刻意反馈循环"：审查上周发生了什么、调整配置、让下一周受益于所学。大多数人对 Claude Code 的使用会在早期平台期就停滞，而持续迭代 Harness 是打破瓶颈的关键。

revfactory — Harness 效果的量化研究

revfactory 团队对 15 个软件工程任务进行了对照实验，量化了 Harness 对 LLM 代码 Agent 输出质量的影响。核心发现：Harness 效果随任务复杂度指数增长——基础任务提升 +23.8 分，高级任务提升 +29.6 分，专家级任务提升 +36.2 分。这证实了 Harness 的价值不在简单任务上，而在复杂的、长期的、多步骤的项目中。

Manus — 五次重构的教训

Manus 团队经历了五次 Harness 重构，总结出"可撕裂原则"（Rippable Harness）：随着模型能力增强，当前 Harness 中的部分组件会被模型本身吸收，因此 Harness 应该设计为可拆除的脚手架——当某个约束不再需要时，能够干净地移除而不影响系统其他部分。

Boris Cherny — 计划与执行分离

Boris Cherny 在 CLAUDE.md 中设置了明确规则："对任何非琐碎任务（3 步以上或涉及架构决策）进入计划模式。如果出了问题，立即停止并重新规划——不要硬推。" 他还在 2026 年 1 月引入了一个关键优化：用户接受计划后，Claude Code 自动清除上下文，让执行阶段从干净的上下文开始。研究和执行分离实际上确保了计划被更准确地实现。

Geoffrey Huntley — Ralph Wiggum Loop 与反压机制

独立开发者 Geoffrey Huntley 提出了 Ralph Wiggum Loop 概念——Agent 在循环中自主验证、自我修复、直到所有审查条件满足的自治闭环。OpenAI 在其报告中也采用了这一术语。同时引入 Backpressure（反压）概念：当 Agent 的代码产出速度超过人类审查能力时，系统需要自动降速，避免低质量代码大量堆积。核心洞察是——无约束的自主性不是自由，而是混乱。

Dex Horthy — 上下文窗口利用率甜蜜区

Dex Horthy 通过大量实验提出了上下文窗口利用率的分区模型，并设计了 Research-Plan-Implement 工作流：

利用率区间状态表现
0-15%Starved Zone上下文不足，Agent 缺乏必要信息
15-40%Smart Zone（甜蜜区）Agent 表现最佳，信息充足且不干扰
>40%Dumb Zone幻觉、循环、格式错误的工具调用、低质量代码

核心启示：给 Agent 塞一堆 MCP 工具、冗长文档和累积的对话历史，不会让它更聪明——反而会让它变笨。他提出 Frequent Intentional Compaction（频繁有意压缩）作为管理上下文膨胀的关键技术。

LangChain — Agent Harness 解剖学

LangChain 在博客《The Anatomy of an Agent Harness》中系统化定义了 Harness 的组成："Agent = Model + Harness"——Harness 是模型之外的一切代码、配置和执行逻辑。他们验证了 Opus 4.6 在 Claude Code 中的得分远低于在其他 Harness 中的表现，证明模型与 Harness 的匹配度比模型本身更重要。他们的 deepagents-cli 通过仅在 Harness 中加入更激进的验证提示，得分跃升 13.7 个百分点。

Vasilopoulos 等 — 三层上下文架构（学术验证）

基于 283 个开发会话、108K 行 C# 代码库的学术研究，提出并验证了三层上下文基础设施：Hot Memory（热记忆：当前任务直接相关的文件和指令）、Domain Experts（领域专家：按需加载的 19 个专门化知识模块）、Cold Memory（冷记忆：历史决策、完成的计划、技术债记录）。该研究提供了 Harness Engineering 在学术层面的定量证据。

多源实践总结

所有团队独立发现的共同模式：前置研究与规划（effort 应前移到规划阶段）、验证输出静默化（只输出错误）、上下文隔离（子 Agent 保持主上下文干净）、按失败驱动迭代（不预防性优化）。

十、落地实践指南

从今天开始的三件事

📝

1. 维护 AGENTS.md

创建一份随 Agent 失败案例持续更新的 AGENTS.md。每当 Agent 犯错，补充一条规则。控制在 ~100 行，作为地图索引而非百科全书。

🔧

2. 编码架构约束为 Linter

把关键架构规则写成 Linter 规则而不是文档。Linter 错误消息包含修复指引，让 Agent 自我纠正。

🔄

3. CI 自动重试

给 Agent 的 CI 流程加上"带错误信息的自动重试"钩子。Agent 读取错误信息后自行修复，循环直到通过。

🧹

4. 定期清洁任务

每周运行一次清理 Agent，检查文档和代码的一致性、架构规范违规、可抽象的重复模式。

十一、用 Claude Code 实现 Harness Engineering

Claude Code 是 Anthropic 推出的命令行 Agent 工具，其自身的架构就是一个生产级 Harness 的典型实现：内置权限模型、Hooks 系统、子 Agent 隔离、渐进式披露的 Skills 体系。以下详细解析如何利用 Claude Code 的原生能力构建完整的 Harness。

11.1 CLAUDE.md — 信息层的核心入口

CLAUDE.md 是 Claude Code 的 AGENTS.md 等价物，是 Agent 进入项目时看到的第一份指南。Claude Code 支持三级 CLAUDE.md 层次结构：

层级位置作用域内容
全局~/.claude/CLAUDE.md所有项目个人偏好、通用编码风格、全局规则
项目项目根/CLAUDE.md当前项目项目架构、构建命令、约束规则、已知陷阱
子目录src/feature/CLAUDE.md特定模块模块级别的局部规则覆盖
CLAUDE.md 最佳实践

控制在 60-100 行。做地图索引，不做百科全书。用 # 前缀可以直接在终端中快速添加规则到 CLAUDE.md，无需手动编辑文件。

11.2 Hooks — 约束层的确定性执行

Hooks 是 Claude Code 实现 Harness 约束的核心机制。与 CLAUDE.md 中的 prompt 指令不同，Hooks 是确定性执行的——每次都以完全相同的方式运行，不受模型"灵活解读"的影响。

Claude Code 支持 21 种生命周期事件和 4 种处理器类型（command / http / prompt / agent），覆盖从会话启动到结束的完整链路。

核心 Hook 事件与 Harness 映射

事件触发时机Harness 用途
PreToolUse工具执行前安全检查、权限验证、危险操作拦截
PostToolUse工具执行后自动格式化、类型检查、Lint 运行
StopAgent 完成响应自动测试、PR 检查、质量验证
SubagentStop子 Agent 完成监控子 Agent 输出质量
SessionStart会话启动环境变量注入、开发上下文加载
PermissionRequest请求权限时自动批准安全操作、拦截危险操作

实战 Hook 配置示例

// ~/.claude/settings.json 或 .claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "./scripts/safety-check.sh"
            // 拦截危险命令：rm -rf、git push --force 等
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write|Edit|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "npx prettier --write \"$FILE_PATH\" 2>/dev/null; exit 0"
            // 文件修改后自动格式化
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "npm test 2>&1 | tail -20"
            // Agent 完成后自动跑测试，只输出最后 20 行
          }
        ]
      }
    ]
  }
}
Hook 设计原则

成功静默，只输出错误。不要让 4000 行通过的测试结果淹没上下文窗口。验证通过时返回空输出，失败时才提供详细错误信息——这是 HumanLayer 用血泪换来的经验。

11.3 Sub-Agents 与 Worktrees — 上下文隔离

Sub-Agent 是 Claude Code 实现 Harness 中"上下文隔离"的核心能力。每个 Sub-Agent 拥有独立的上下文窗口，主 Agent 只看到 Sub-Agent 的最终摘要，而非完整的工作过程。

Sub-Agent 架构模式

# .claude/agents/code-reviewer.md
---
name: code-reviewer
description: 审查代码质量、安全性和可维护性
tools: ["Read", "Grep", "Glob", "Bash"]
model: opus
---
你是一位资深代码审查者。审查时关注：
1. 是否违反架构约束（参考 docs/architecture.md）
2. 是否存在安全漏洞
3. 测试覆盖是否充分
4. 是否引入技术债
以结构化格式输出审查结果。
# .claude/agents/doc-gardener.md
---
name: doc-gardener
description: 扫描文档与代码的不一致
tools: ["Read", "Grep", "Glob", "Write"]
model: sonnet
background: true
---
你是文档园丁 Agent。扫描 docs/ 目录，对比实际代码：
1. 找出过时的 API 文档
2. 检查架构图是否反映当前代码结构
3. 验证 CLAUDE.md 中的命令是否仍然有效
发现问题后直接修复并提交。

Git Worktrees 并行开发：使用 claude --worktree 或在 Agent 定义中设置 isolation: "worktree"，Claude Code 会创建隔离的 git worktree 副本。多个 Agent 可以同时在不同 worktree 中工作，互不干扰——这是实现 Agent 并行开发的基础设施。

# 终端 1：主功能开发
claude --worktree "实现用户认证模块"

# 终端 2：并行修 Bug
claude --worktree "修复支付回调超时问题"

# 终端 3：后台文档维护
claude --worktree "运行文档园丁 Agent"

11.4 Skills — 渐进式披露的能力包

Skills 是 Claude Code 实现"渐进式披露"的机制——只在特定任务出现时才加载对应的方法论，避免在上下文中塞入不相关的指令。

# .claude/skills/tdd/SKILL.md
---
name: tdd-workflow
description: 当需要实现新功能或修复 Bug 时触发
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/check-test-first.sh"
---
## TDD 工作流
1. 先定义接口
2. 写失败的测试（RED）
3. 写最小实现让测试通过（GREEN）
4. 重构（REFACTOR）
5. 提交前确认所有测试通过

## 参考
- 测试规范: references/test-conventions.md
- 示例: examples/auth-tdd-example.md
Skills 设计要点

Skills 是文件夹，不是文件。使用 references/、scripts/、examples/ 子目录实现渐进式披露。Skill 的 description 字段是触发器，不是摘要——要为模型写（"什么时候该触发我？"）。在每个 Skill 中建立 Gotchas 部分，记录 Claude 的历史失败点。

11.5 Slash Commands — 日常工作流自动化

如果你每天做某件事超过一次，就应该把它变成一个 Slash Command。Commands 存放在 .claude/commands/ 目录中，纳入 git 版本控制。

# .claude/commands/techdebt.md
---
description: 扫描并修复技术债
---
扫描代码库中的技术债：
1. 运行 `grep -rn "TODO\|FIXME\|HACK" src/` 统计标记
2. 检查 docs/quality.md 中的质量评分
3. 识别最高优先级的 3 个技术债项
4. 对每一项生成修复 PR
使用子 Agent 并行处理不同的修复项。
# .claude/commands/review.md
---
description: 提交前的完整审查流程
---
执行提交前审查：
1. 运行 `git diff --stat` 查看变更范围
2. 启动 code-reviewer Agent 审查代码质量
3. 运行完整测试套件
4. 检查是否需要更新文档
5. 验证是否违反架构约束
只有所有检查通过后，才生成 PR 描述。

11.6 settings.json — 确定性行为配置

对于需要确定性执行的行为，使用 settings.json 而非 CLAUDE.md。CLAUDE.md 中的指令是"建议"，settings.json 中的配置是"强制"。

// .claude/settings.json（项目级别）
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(npx prettier *)",
      "Bash(git status)",
      "Bash(git diff *)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force *)",
      "Bash(curl * | bash)"
    ]
  },
  "attribution": {
    "commit": ""   // 不添加 Co-Authored-By（用 settings 强制，不用 CLAUDE.md）
  }
}

十二、业界共识与开放问题

六大共识

综合 OpenAI、Anthropic、LangChain、Stripe、HashiCorp 等多个独立信息源，业界在以下方面已形成明确共识：

共识 1：Harness 不是框架的替代品，而是位于框架之上的一层。传统框架解决"如何构建 AI Agent"，Harness 解决"Agent 如何可靠地运行"。

共识 2：模型正在吸收框架约 80% 的功能（Agent 定义、消息路由、任务生命周期），但剩余 20%——持久化、确定性重放、成本控制、可观测性、错误恢复——正是 Harness 存在的价值。

共识 3：在纠结模型选择之前，先审视 Harness 设计能获得更高的投资回报率。同一模型在不同 Harness 下性能差异巨大。

共识 4：所有团队都独立发现相同的模式——分层上下文、渐进式披露、计划先行、机械化约束。

共识 5：约束是速度的前提。为了获得更高的 AI 自主性，运行时必须受到更严格的约束。增加信任需要的不是更多自由，而是更多限制。

共识 6：工程师角色正在转变——从"代码编写者"变为"环境设计者"。

四个开放争议

议题争议焦点
上下文窗口利用率Dex Horthy 提出 ~40% 为"甜蜜区"上限，超过则出现幻觉和循环。但此阈值是否适用于所有模型和任务类型尚无定论
Harness 寿命随着模型能力增强，当前 Harness 中多少会被模型本身吸收？"可撕裂原则"（Rippable Harness）主张设计可拆除的脚手架
Agent-to-Agent Review 的可靠性Anthropic 指出模型无法可靠评估自身工作，需要分离 Generator 和 Evaluator
棕地项目改造所有公开案例均为 greenfield，10 年历史遗留代码库的 Harness 化路径仍是完全空白

三个待探索空白

1. AI 代码的长期可维护性：当代码的作者不再是人类，而是 Agent 集群，我们如何理解它、维护它、与之共存？

2. Harness 模板的标准化：未来是否会出现类似"服务模板"的 Harness 模板——包含自定义 Linter、结构化测试、基础文档、架构约束规则——供团队快速启动？

3. 跨团队 Harness 协同：当多个团队各自运行不同 Agent 和 Harness 时，如何确保系统级一致性？
